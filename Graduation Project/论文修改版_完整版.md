# 基于Django的校园二手交易系统设计与实现

## 摘要

随着大学生消费水平的提高和个人物品更新换代的加速，校园内二手物品交易需求日益增长。然而，现有的社交媒体群存在信息杂乱、缺乏交易保障的问题，而通用平台又常被校外房源干扰且风险较高。为此，本研究设计并实现了一个基于Django框架的校园二手交易管理系统，旨在构建一个安全、智能、高效的专属平台。

本系统采用Django MTV（Model-Template-View）模式与前后端分离架构，后端基于Django REST框架构建，前端采用响应式设计技术。研究重点聚焦于三个关键领域：构建结构化的用户行为知识库，整合搜索、浏览、收藏、交易记录，为智能服务提供数据基础；提出融合内容特征的个性化推荐算法，引入时间衰减因子缓解"冷启动"问题，提高交易匹配效率；建立多层安全机制，包括校园邮箱认证、数据匿名化与加密，增强交易可信度。

系统实现了用户管理、商品发布、智能搜索、在线交易、信用评价、站内消息等功能模块。测试表明，系统运行稳定，推荐算法效果良好，为促进校园资源循环利用、支持智慧校园发展提供了可行方案。

**关键词：** 校园二手交易；用户行为知识库；推荐算法；安全架构

## Abstract

With the rising consumption level of college students and the accelerated turnover of personal items, the demand for trading idle goods on campus is growing rapidly. However, existing social media groups suffer from cluttered information and lack of transaction safeguards, while general-purpose platforms are often disrupted by off-campus listings and involve higher risks. In response, this study designs and implements a campus second-hand trading management system based on the Django framework, aiming to establish a secure, intelligent, and efficient dedicated platform.

The system adopts the Django MTV (Model-Template-View) pattern with a frontend-backend separation architecture. The backend is built on the Django REST framework, while the frontend employs responsive design techniques. The research focuses on three key areas: constructing a structured user behavior knowledge base that integrates search, browsing, bookmarking, and transaction records to provide a data foundation for intelligent services; proposing a personalized recommendation algorithm that combines item-based collaborative filtering with content features, incorporating a time decay factor to mitigate the "cold start" problem and improve transaction matching efficiency; establishing a multi-layered security mechanism, including campus email authentication, data anonymization, and encryption, to enhance transaction credibility.

The system implements functional modules such as user management, product listing, intelligent search, online transactions, credit evaluation, and in-site messaging. Testing demonstrates that the system operates stably and the recommendation algorithm performs effectively, offering a viable solution for promoting resource recycling on campus and supporting the development of smart campuses.

**Keywords:** Campus Second-Hand Trading; User Behavior Knowledge Base; Recommendation Algorithm; Security Framework

## 1 绪论

### 1.1 研究背景与意义

#### 1.1.1 研究背景

随着高等教育规模扩大，大学生群体积累了大量闲置物品，如教材书籍、电子产品等，由于缺乏有效流通渠道，造成资源浪费。同时，环保意识增强推动二手交易需求增长。

当前校园二手交易主要依赖微信群、QQ群等社交媒体，存在信息混乱、身份验证缺失、交易保障不足、虚假信息泛滥等问题，严重影响交易效率和安全性。因此，构建专门的校园二手交易平台显得尤为必要，需具备完善的用户认证、智能推荐和安全保障机制。

#### 1.1.2 研究意义

本研究在理论和实践层面均具有重要价值。理论方面，将知识图谱技术、推荐算法和安全机制应用于校园二手交易场景，为个性化推荐系统设计提供新方案。实践方面，通过校园邮箱认证确保用户真实性，智能推荐算法提高交易效率，完善交易流程保障双方权益，促进校园资源循环利用。研究成果可推广至社区、企业等垂直领域，具有广泛应用前景。

### 1.2 国内外研究现状

#### 1.2.1 国内研究现状

国内研究主要集中在商业模式、用户行为和技术实现三个方面。商业模式方面，学者们分析了C2C、B2C、C2B2C等模式，指出校园二手交易更适合C2C模式。用户行为方面，研究发现价格优势是大学生选择二手交易的主要原因，质量担忧和安全顾虑是主要障碍。技术实现方面，协同过滤算法广泛应用，信任机制研究提出了基于评价、交易记录的综合评估模型。

然而，现有研究存在理论层面多、实际验证少，推荐算法简单、未充分考虑商品特征和时间因素，安全机制薄弱、难以应对复杂威胁等问题。

#### 1.2.2 国外研究现状

国外研究起步较早，在推荐系统方面提出了矩阵分解、深度学习等先进算法，提高了推荐精准度。安全与隐私保护方面，差分隐私、同态加密等技术得到应用，区块链去中心化交易模式也受关注。可持续发展方面，研究表明二手交易平台能有效减少碳排放、创造经济价值。

但国外成果直接应用于国内校园场景存在局限，需考虑文化差异、使用习惯、监管环境等因素，因此开发适合国内校园特点的平台仍具重要价值。

### 1.3 研究内容与目标

#### 1.3.1 研究内容

本研究围绕校园二手交易平台的设计与实现展开，主要包括用户行为知识库构建、个性化推荐算法设计、安全架构建立和系统功能模块实现四个方面。

用户行为知识库构建通过设计合理的数据模型，采集和存储用户浏览、搜索、收藏、交易等行为数据，采用分层架构为推荐算法和安全决策提供数据支撑。个性化推荐算法综合考虑浏览量、收藏数、交易状态、时间衰减等因素，引入时间衰减因子缓解"冷启动"问题，采用混合推荐策略为不同用户提供合适推荐。

安全架构采用三层防护策略：身份安全层通过校园邮箱认证确保用户真实性；行为安全层通过异常检测防范恶意操作；数据安全层通过加密技术保护用户数据。系统功能模块包括用户管理、商品管理、推荐系统、交易管理、消息系统、安全管理等，覆盖完整业务流程。

#### 1.3.2 研究目标

本研究旨在开发功能完善、性能优良、安全可靠的校园二手交易平台，具体目标包括：构建结构化用户行为知识库，支持高效查询和扩展；设计多维度推荐算法，综合考虑商品特征、用户偏好、时间因素，推荐响应时间控制在500毫秒内；建立多层安全防护体系，识别防范常见威胁，加密存储传输敏感数据；实现完整交易功能，界面简洁友好，操作易懂易用。

## 2 相关技术介绍

### 2.1 Django框架

#### 2.1.1 Django概述

Django是开源的高级Python Web框架，遵循DRY原则，提供ORM、表单处理、用户认证等丰富功能，强调快速开发、安全可靠。内置多种安全防护机制，社区活跃，文档完善，是Web开发主流框架之一。

#### 2.1.2 MTV架构模式

Django采用MTV（Model-Template-View）架构模式，将应用程序分为模型（数据层）、模板（表现层）、视图（业务逻辑层）三个核心组件。模型负责定义数据结构和验证规则，模板负责界面展示，视图负责处理请求和业务逻辑。各层职责清晰，便于团队协作和独立开发测试。

#### 2.1.3 ORM系统原理

ORM（对象关系映射）是Django核心组件，将数据库表映射为Python类，提供面向对象的数据库操作方式。工作原理包括模型定义、查询构建、SQL生成三个阶段。ORM优势在于提高开发效率，提供数据库抽象层，自动处理SQL注入防护，支持多种数据库后端切换。

#### 2.1.4 Django的优势

Django具有开发效率高（丰富内置功能、自动生成管理后台）、安全性好（内置多种防护机制）、可扩展性强（松耦合设计、中间件支持）、文档完善（详尽官方文档、活跃社区）等优势，适合企业级应用开发。

### 2.2 前端技术

#### 2.2.1 Bootstrap响应式设计

Bootstrap是由Twitter开发的开源前端框架，是目前最流行的HTML、CSS和JavaScript框架之一。Bootstrap提供了丰富的预定义样式和组件，使开发者能够快速构建美观、响应式的Web界面。

响应式设计是Bootstrap的核心特性。Bootstrap采用移动优先的设计理念，通过网格系统实现不同屏幕尺寸下的自适应布局。网格系统将页面分为12列，开发者可以通过指定列数来控制元素在不同屏幕尺寸下的显示方式。Bootstrap定义了四种屏幕尺寸断点：超小屏幕（<576px）、小屏幕（≥576px）、中等屏幕（≥768px）、大屏幕（≥992px）、超大屏幕（≥1200px），针对不同断点可以设置不同的布局规则。

Bootstrap提供了丰富的UI组件，包括导航栏、按钮、表单、卡片、模态框、轮播图等。这些组件具有一致的设计风格和交互效果，可以直接使用或根据需要进行定制。Bootstrap的组件基于HTML、CSS和JavaScript实现，不依赖其他JavaScript框架，可以与任何前端技术栈配合使用。

Bootstrap的样式系统基于Sass预处理器构建，提供了大量的CSS变量和工具类。开发者可以通过修改变量来定制主题颜色、字体、间距等样式属性。工具类提供了快速设置边距、填充、文本对齐、显示属性等样式的方法，减少了自定义CSS的编写。

本系统采用Bootstrap 5版本，这是Bootstrap的最新稳定版本。Bootstrap 5放弃了对jQuery的依赖，使用原生JavaScript实现组件交互，减少了项目的依赖体积。Bootstrap 5还引入了新的组件和工具类，改进了网格系统和表单控件，提供了更好的开发体验。

#### 2.2.2 Font Awesome图标库

Font Awesome是一个流行的图标库，提供了数千个矢量图标，可用于Web和桌面应用。Font Awesome的图标基于字体实现，可以像文本一样设置大小、颜色、阴影等样式属性，具有无损缩放、轻量级、易于使用的特点。

Font Awesome提供了多种图标类别，包括Web应用图标、表单控件图标、支付图标、图表图标、品牌图标等，覆盖了常见的使用场景。图标采用统一的风格设计，保证了界面的一致性。Font Awesome还支持图标动画、图标堆叠、图标旋转等高级功能，增强了图标的表现力。

Font Awesome的使用非常简单。开发者只需引入Font Awesome的CSS文件，然后在HTML中使用特定的类名即可显示图标。例如，使用`<i class="fas fa-user"></i>`可以显示用户图标。Font Awesome 6引入了SVG图标支持，提供了更清晰的图标渲染效果。

本系统使用Font Awesome 6图标库，为界面提供丰富的图标元素。图标用于导航菜单、按钮、表单提示、状态指示等场景，增强了界面的直观性和美观性。

#### 2.2.3 JavaScript交互技术

JavaScript是Web开发的核心语言，用于实现网页的动态交互效果。本系统使用原生JavaScript实现前端交互功能，不依赖大型JavaScript框架，保持了项目的轻量级和简洁性。

DOM操作是JavaScript的基础功能，用于动态修改网页的内容、结构和样式。本系统使用原生DOM API进行元素选择、内容修改、样式设置、事件绑定等操作。例如，使用`document.getElementById`选择元素，使用`element.addEventListener`绑定事件监听器，使用`element.classList`操作CSS类名。

表单验证是JavaScript的重要应用场景。本系统在客户端进行表单验证，提供即时的错误提示，改善用户体验。验证逻辑包括必填字段检查、格式验证（如邮箱格式、手机号格式）、长度验证、一致性验证（如密码和确认密码）等。验证通过后才提交表单到服务器，减少了无效请求。

AJAX通信是JavaScript的核心功能，用于实现页面的局部刷新。本系统使用Fetch API进行异步数据请求，实现无刷新表单提交、动态内容加载、实时搜索等功能。Fetch API基于Promise，提供了更清晰的异步编程模型，相比传统的XMLHttpRequest更加简洁易用。

#### 2.2.4 AJAX异步通信

AJAX（Asynchronous JavaScript and XML，异步JavaScript和XML）是一种在无需重新加载整个页面的情况下，与服务器交换数据并更新部分网页的技术。AJAX使Web应用能够提供更加流畅和响应迅速的用户体验。

AJAX的工作原理是通过JavaScript创建XMLHttpRequest对象或使用Fetch API，向服务器发送异步请求。服务器处理请求后返回数据，通常是JSON格式。JavaScript接收到响应数据后，使用DOM操作更新页面的相应部分，无需刷新整个页面。

本系统广泛使用AJAX技术提升用户体验。在商品搜索功能中，用户输入关键词后，系统通过AJAX请求获取搜索结果，动态更新商品列表，无需页面跳转。在消息系统中，新消息通过AJAX实时获取并显示，提供即时通讯的体验。在表单提交中，AJAX用于异步验证和提交，提供即时反馈。

Fetch API是现代浏览器提供的AJAX实现方式，相比XMLHttpRequest更加简洁和强大。Fetch API基于Promise，支持async/await语法，使异步代码更加清晰易读。Fetch API还提供了更灵活的配置选项，如请求头设置、跨域处理、请求取消等。

### 2.3 数据库技术

#### 2.3.1 SQLite数据库特性

SQLite是一个轻量级的嵌入式关系型数据库管理系统，具有零配置、无服务器、单文件存储等特点。SQLite广泛应用于移动应用、桌面软件、小型Web应用等场景，是开发阶段和小型应用的理想选择。

零配置是SQLite的显著特点。SQLite不需要安装、配置或管理，只需引入库文件即可使用。没有独立的服务器进程，数据库操作直接通过函数调用完成。这大大简化了数据库的部署和维护工作，降低了使用门槛。

单文件存储是SQLite的存储特性。整个数据库存储在一个磁盘文件中，包括表、索引、视图、触发器等所有数据库对象。这种存储方式便于数据库的备份、迁移和共享。只需复制数据库文件，即可完成数据库的备份和迁移。

轻量级是SQLite的体积特点。SQLite库文件体积小，完整功能库仅约1MB，裁剪版本可更小。运行时内存占用低，适合资源受限的环境。尽管轻量，SQLite支持大部分SQL92标准，提供了完整的关系型数据库功能。

事务支持是SQLite的可靠性保证。SQLite支持ACID事务，确保数据操作的原子性、一致性、隔离性和持久性。即使在系统崩溃或断电情况下，SQLite也能保证数据的完整性。SQLite还支持多种事务隔离级别，满足不同应用场景的需求。

本系统选择SQLite作为开发阶段的数据库，主要考虑其部署简单、使用方便的特点。对于生产环境，可以根据用户规模迁移到MySQL或PostgreSQL等更强大的数据库系统。

#### 2.3.2 关系型数据库设计

关系型数据库是基于关系模型的数据库管理系统，使用表（Table）来组织数据，表由行（Row）和列（Column）组成。关系型数据库具有数据结构清晰、数据一致性高、查询功能强大等特点，是目前应用最广泛的数据库类型。

数据库设计是应用开发的重要环节，良好的数据库设计能够提高数据存储效率、保证数据完整性、支持高效的数据查询。数据库设计通常遵循规范化原则，通过分解表结构消除数据冗余，避免数据更新异常。

第一范式（1NF）要求表的每个列都是原子的，不可再分。每个单元格只包含一个值，不存在重复的列组。第二范式（2NF）在满足1NF的基础上，要求非主键列完全依赖于主键，不存在部分依赖。第三范式（3NF）在满足2NF的基础上，要求非主键列直接依赖于主键，不存在传递依赖。

在实际设计中，需要在规范化和性能之间取得平衡。过度规范化可能导致表数量过多，查询时需要大量连接操作，影响性能。适当的反规范化可以减少连接操作，提高查询效率，但会增加数据冗余和维护复杂度。

本系统的数据库设计遵循第三范式，确保数据的完整性和一致性。对于频繁查询的场景，通过建立适当的索引和视图来优化查询性能。对于需要频繁连接查询的数据，考虑适度的反规范化设计。

#### 2.3.3 索引优化策略

索引是数据库中用于提高查询效率的数据结构，类似于书籍的目录。通过建立索引，数据库可以快速定位到满足查询条件的数据，避免全表扫描，大大提高查询速度。

索引的类型主要包括B树索引、哈希索引、全文索引等。B树索引是最常用的索引类型，适用于等值查询和范围查询。哈希索引适用于等值查询，查询速度极快，但不支持范围查询。全文索引用于文本内容的搜索，支持关键词匹配和相关性排序。

索引的设计需要考虑查询模式和数据更新频率。对于频繁查询的列应该建立索引，对于频繁更新的列应该谨慎建立索引，因为索引的维护会增加更新操作的开销。复合索引可以用于多列查询条件，但列的顺序很重要，应该将选择性高的列放在前面。

索引并非越多越好。过多的索引会占用额外的存储空间，增加数据更新的开销，降低写入性能。因此，应该根据实际的查询需求，建立必要的索引，避免过度索引。

本系统根据查询需求设计了合理的索引策略。对于主键、外键、唯一约束自动建立索引。对于频繁查询的字段，如用户名、邮箱、商品状态等，建立单列索引或复合索引。对于文本搜索，考虑使用全文索引或专门的搜索引擎。

### 2.4 关键技术选型

#### 2.4.1 技术选型原则

技术选型是系统开发的重要决策，直接影响开发效率、系统性能和维护成本。本系统的技术选型遵循以下原则。

适用性原则。选择的技术应该适合项目的实际需求，能够解决项目面临的核心问题。不盲目追求新技术，也不过时守旧，选择成熟稳定、生态完善的技术栈。

效率性原则。选择的技术应该能够提高开发效率，减少重复工作。优先选择具有丰富内置功能和第三方库的技术，避免从零开发通用功能。同时，技术栈应该便于团队协作，降低沟通成本。

性能性原则。选择的技术应该能够满足系统的性能需求，包括响应时间、并发处理能力、资源占用等方面。对于性能关键的场景，选择经过优化的技术方案。

安全性原则。选择的技术应该具有良好的安全特性，能够帮助构建安全的应用。优先选择提供安全机制、及时修复漏洞、社区活跃的技术。

可维护性原则。选择的技术应该便于后期维护和升级。优先选择文档完善、社区活跃、版本更新稳定的技术。避免使用过于小众或即将淘汰的技术。

#### 2.4.2 技术栈对比分析

后端框架方面，主要考虑了Django、Flask、Spring Boot三个选项。Django是Python的全栈框架，功能丰富，适合快速开发。Flask是Python的微框架，轻量灵活，适合小型项目。Spring Boot是Java的框架，性能优异，适合大型企业级应用。考虑到开发效率和Python的简洁性，本系统选择Django作为后端框架。

前端框架方面，主要考虑了Bootstrap、Vue.js、React三个选项。Bootstrap是CSS框架，提供现成的UI组件，适合快速构建界面。Vue.js和React是JavaScript框架，适合构建复杂的单页应用。考虑到项目的复杂度和开发周期，本系统选择Bootstrap作为前端框架，使用原生JavaScript实现交互。

数据库方面，主要考虑了SQLite、MySQL、PostgreSQL三个选项。SQLite轻量简单，适合开发阶段和小型应用。MySQL和PostgreSQL功能强大，适合生产环境。考虑到开发便利性，本系统选择SQLite作为开发数据库，保留迁移到MySQL或PostgreSQL的可能性。

#### 2.4.3 最终技术方案

综合以上分析，本系统采用以下技术方案。

后端采用Django框架，利用其丰富的内置功能和良好的安全特性，快速构建后端服务。使用Django ORM进行数据库操作，使用Django模板引擎渲染页面，使用Django中间件处理请求和响应。

前端采用Bootstrap 5实现响应式布局，使用Font Awesome 6提供图标，使用原生JavaScript实现交互效果，使用Fetch API进行AJAX异步通信。

数据库采用SQLite，利用其零配置、单文件存储的特点，简化开发和部署。未来可以根据需要迁移到MySQL或PostgreSQL。

开发工具使用Visual Studio Code作为代码编辑器，使用Git进行版本控制，使用venv管理Python虚拟环境，使用Django Test Framework进行测试。

部署采用Docker容器化方案，通过Docker Compose编排服务，实现开发环境和生产环境的一致性。Web服务器使用Nginx，应用服务器使用Gunicorn。

这套技术方案具有开发效率高、学习成本低、性能出色、可扩展性强、安全性好等优点，能够满足校园二手交易平台的开发需求。

## 3 系统需求分析

### 3.1 系统功能需求

#### 3.1.1 用户管理需求

用户管理是校园二手交易平台的基础功能，负责用户的注册、登录、个人信息管理、密码修改等功能。用户注册功能要求用户通过校园邮箱进行注册，注册时需要填写用户名、密码、确认密码、邮箱等信息。系统需要验证邮箱域名是否为校园邮箱，验证密码强度是否足够，验证两次输入的密码是否一致，验证用户名和邮箱是否已被注册。注册成功后，系统发送验证邮件到用户邮箱，用户需要点击验证链接激活账户。

用户登录功能要求用户通过用户名和密码登录系统。系统需要验证用户名和密码是否正确，如果验证失败，返回错误提示；如果验证成功，记录登录日志，创建用户会话，跳转到首页或用户之前访问的页面。系统还应该支持"记住我"功能，用户下次访问时可以自动登录。用户登出功能要求用户可以随时登出系统。登出时，系统清除用户会话，记录登出日志，跳转到登录页面。

个人信息管理功能要求用户可以查看和编辑个人信息，包括昵称、头像、手机号、个人简介等。用户上传头像时，系统需要对图片进行验证，如格式、大小等，确保图片安全合法。密码修改功能要求用户可以修改密码，修改时需要输入旧密码、新密码、确认新密码。系统需要验证旧密码是否正确，验证新密码强度是否足够，验证两次输入的新密码是否一致。修改成功后，系统提示用户重新登录。

密码找回功能要求如果用户忘记密码，可以通过邮箱找回密码。用户输入注册邮箱，系统发送密码重置链接到邮箱，用户点击链接后可以设置新密码。用户认证功能要求系统需要对用户进行认证，确保只有登录用户才能访问需要权限的页面。对于不同角色的用户（普通用户、管理员），系统需要进行权限控制，确保用户只能访问自己有权限的功能。

#### 3.1.2 商品管理需求

商品管理是校园二手交易平台的核心功能，负责商品的发布、编辑、下架、搜索、筛选等功能。商品发布功能要求登录用户可以发布二手商品，发布时需要填写商品标题、商品描述、商品分类、商品价格、商品图片等信息。系统需要对用户输入进行验证，确保必填字段已填写，价格格式正确，图片格式和大小符合要求。发布成功后，商品状态为"可交易"，显示在商品列表中。

商品编辑功能要求商品发布者可以编辑已发布的商品信息，包括标题、描述、价格、图片等。编辑时，系统需要验证用户是否为商品发布者，只有发布者才能编辑商品。编辑成功后，更新商品信息，记录编辑时间。商品下架功能要求商品发布者可以下架已发布的商品，下架后商品状态为"已下架"，不再显示在商品列表中。系统需要验证用户是否为商品发布者，只有发布者才能下架商品。

商品删除功能要求商品发布者可以删除已发布的商品，删除后商品从数据库中移除。系统需要验证用户是否为商品发布者，只有发布者才能删除商品。删除操作不可逆，需要用户确认。商品详情功能要求用户可以查看商品的详细信息，包括标题、描述、价格、图片、发布者、发布时间、浏览量等。查看商品详情时，系统记录浏览历史，增加商品的浏览量。

商品搜索功能要求用户可以通过关键词搜索商品，系统根据商品标题和描述进行匹配，显示搜索结果。搜索功能应该支持模糊匹配，用户输入部分关键词也能找到相关商品。商品筛选功能要求用户可以通过分类、价格区间、发布时间等条件筛选商品，系统根据筛选条件显示符合条件的商品。筛选功能应该支持多条件组合，用户可以同时选择分类和价格区间进行筛选。商品列表功能要求系统显示商品列表，商品按发布时间倒序排列，最新发布的商品排在前面。商品列表应该支持分页，每页显示一定数量的商品，避免页面加载过慢。

#### 3.1.3 推荐系统需求

推荐系统是校园二手交易平台的特色功能，负责为用户推荐可能感兴趣的商品，提高交易匹配效率。热门推荐功能要求系统在首页展示热门商品推荐，热门商品根据浏览量、收藏数、交易状态、发布时间等多维度因素计算得出。热门推荐应该定期更新，确保推荐的时效性。

个性化推荐功能要求系统在首页展示"猜你喜欢"个性化推荐，根据用户的历史浏览行为、收藏行为、交易行为等数据，分析用户的偏好，推荐用户可能感兴趣的商品。对于新用户，由于没有历史行为数据，可以采用热门推荐作为冷启动策略。相关商品推荐功能要求系统在商品详情页展示相关商品推荐，推荐与当前商品相似的其他商品。相关商品可以根据分类、价格、标签等因素计算得出，帮助用户发现更多感兴趣的商品。

推荐记录功能要求系统记录推荐结果和用户的反馈，如用户是否点击了推荐商品、是否收藏了推荐商品、是否购买了推荐商品等。通过分析用户反馈，不断优化推荐算法，提高推荐的精准度。推荐解释功能要求系统可以为推荐结果提供简单的解释，如"因为您浏览过XX类商品"、"因为您收藏过XX商品"等，增加推荐的可信度和透明度。

#### 3.1.4 交易管理需求

交易管理是校园二手交易平台的重要功能，负责创建交易、确认交易、完成交易、取消交易等功能。创建交易功能要求买家可以对感兴趣的商品创建交易，创建时需要选择交易方式、交易地点、联系电话等信息。系统需要验证商品状态是否为"可交易"，只有可交易的商品才能创建交易。创建成功后，商品状态变为"交易中"，通知卖家和买家。

确认交易功能要求卖家可以确认买家创建的交易，确认后交易正式开始。系统需要验证用户是否为商品发布者，只有发布者才能确认交易。确认成功后，通知买家交易已确认。完成交易功能要求交易双方完成线下交易后，可以在系统中标记交易完成。标记完成后，交易状态变为"已完成"，商品状态变为"已售出"，交易双方可以互相评价。系统需要验证用户是否为交易双方，只有交易双方才能标记交易完成。

取消交易功能要求交易双方可以在交易完成前取消交易，取消后交易状态变为"已取消"，商品状态恢复为"可交易"。系统需要验证用户是否为交易双方，只有交易双方才能取消交易。取消操作需要填写原因，并通知对方。交易列表功能要求用户可以查看自己的交易列表，包括作为买家的交易和作为卖家的交易。交易列表应该按交易状态和时间分类显示，方便用户查看。

交易详情功能要求用户可以查看交易的详细信息，包括商品信息、交易双方、交易方式、交易地点、交易状态、交易时间等。交易评价功能要求交易完成后，买家和卖家可以互相评价，评价内容包括评分和评论。系统需要验证评价内容是否合规，防止恶意评价。评价信息将作为用户信用档案的重要组成部分，影响用户的信誉度。

#### 3.1.5 消息系统需求

消息系统是校园二手交易平台的辅助功能，负责用户之间的沟通和系统通知。发送消息功能要求用户可以向其他用户发送站内消息，消息内容包括文字和图片。系统需要验证消息内容是否合规，防止发送违规内容。

查看消息功能要求用户可以查看收到的消息列表，消息按时间倒序排列。系统应该区分已读消息和未读消息，未读消息应该有明显的标识。标记已读功能要求用户可以将未读消息标记为已读，也可以将已读消息标记为未读。

删除消息功能要求用户可以删除不需要的消息，删除后消息从数据库中移除。系统通知功能要求系统可以向用户发送系统通知，如交易状态变更通知、商品下架通知、安全警告通知等。系统通知应该有特殊的标识，与普通消息区分开。

#### 3.1.6 安全管理需求

安全管理是校园二手交易平台的重要保障，负责用户身份验证、异常行为检测、数据安全保护等功能。身份验证功能要求系统需要验证用户的身份，确保用户身份的真实性。身份验证包括校园邮箱域名验证、密码强度验证、双因素认证等。

异常登录检测功能要求系统需要检测异常登录行为，如短时间内频繁登录、异地登录、使用陌生设备登录等。检测到异常登录时，系统应该发出警告，要求用户进行额外验证，如邮箱验证、手机验证等。异常行为监控功能要求系统需要监控用户的异常行为，如频繁发布违规商品、恶意评价、频繁发送垃圾消息等。检测到异常行为时，系统应该记录日志，并根据严重程度采取相应措施，如警告、限制功能、封禁账号等。

数据安全保护功能要求系统需要保护用户数据的安全，包括密码加密存储、敏感数据加密传输、输入验证防止XSS攻击和SQL注入等。安全审计功能要求系统需要记录安全相关的操作，如登录、登出、密码修改、权限变更等。安全日志应该包括操作时间、操作类型、操作用户、操作结果等信息，便于事后追溯和分析。

### 3.2 系统非功能需求

#### 3.2.1 性能需求

性能需求是指系统在响应时间、吞吐量、并发处理能力等方面需要满足的要求。响应时间方面，系统的响应时间应该控制在合理范围内，确保用户体验的流畅性。首页加载时间应该控制在2秒以内，商品列表页面加载时间应该控制在3秒以内，商品详情页面加载时间应该控制在2秒以内，推荐算法的响应时间应该控制在500毫秒以内。

并发处理能力方面，系统应该能够支持一定规模的并发用户访问，确保在高峰期也能稳定运行。系统应该能够支持至少100个并发用户同时访问，在100个并发用户的情况下，系统的响应时间不应该超过3秒。吞吐量方面，系统应该能够处理一定数量的请求，确保业务的正常进行。系统应该能够支持每秒至少处理50个请求，在处理高峰期也能保持稳定的性能。

数据库性能方面，数据库查询应该高效，确保数据的快速存取。简单查询的响应时间应该控制在100毫秒以内，复杂查询的响应时间应该控制在500毫秒以内。

#### 3.2.2 可用性需求

可用性需求是指系统在可访问性、易用性、兼容性等方面需要满足的要求。可访问性方面，系统应该保证7×24小时的可用性，除了计划内的维护时间外，系统应该始终可用。系统的可用性应该达到99.5%以上，全年非计划停机时间不超过43.8小时。

易用性方面，系统的界面设计应该简洁直观，操作流程应该简单易懂，用户能够快速上手使用。系统应该提供清晰的导航和操作提示，减少用户的学习成本。兼容性方面，系统应该兼容主流的浏览器和操作系统，确保在不同设备上都能正常使用。系统应该支持Chrome、Firefox、Safari、Edge等主流浏览器，支持Windows、macOS、Linux等操作系统，支持PC端、平板和手机等不同设备。

响应式设计方面，系统应该采用响应式设计，确保在不同屏幕尺寸的设备上都能良好显示。系统应该能够自适应不同的屏幕分辨率，在手机、平板和桌面电脑上都能提供良好的用户体验。

#### 3.2.3 安全性需求

安全性需求是指系统在数据安全、身份安全、行为安全等方面需要满足的要求。数据安全方面，用户的敏感数据应该加密存储，如密码应该使用哈希算法加密存储，不能明文存储。敏感数据在传输过程中应该加密，使用HTTPS协议进行传输。系统应该定期备份数据，防止数据丢失。

身份安全方面，系统应该验证用户身份的真实性，防止恶意注册和冒充。用户注册时应该验证邮箱域名，确保是校园邮箱。密码应该有强度要求，至少8位，包含大小写字母、数字和特殊字符。系统应该支持双因素认证，提高账户的安全性。行为安全方面，系统应该检测和防范异常行为，如异常登录、恶意评价、垃圾消息等。检测到异常行为时，系统应该及时发出警告，并采取相应措施。系统应该记录用户的操作日志，便于事后追溯和分析。

防攻击方面，系统应该防范常见的Web攻击，如XSS攻击、SQL注入、CSRF攻击、点击劫持等。系统应该对用户输入进行验证和消毒，防止恶意输入。系统应该使用CSRF令牌，防止CSRF攻击。

#### 3.2.4 可扩展性需求

可扩展性需求是指系统在功能扩展、性能扩展、技术扩展等方面需要满足的要求。功能扩展方面，系统的架构设计应该模块化，便于未来添加新的功能。各个功能模块之间应该低耦合，修改一个模块不会影响其他模块。系统应该预留扩展接口，便于未来集成第三方服务。

性能扩展方面，系统应该能够根据用户规模的增长进行性能扩展，如增加服务器、使用缓存、使用消息队列等。系统的数据库设计应该支持水平扩展，如分库分表等。技术扩展方面，系统的技术选型应该考虑未来的技术发展，便于未来进行技术升级。系统应该能够方便地从SQLite迁移到MySQL或PostgreSQL，从单机部署迁移到分布式部署。

知识库和推荐算法扩展方面，知识库和推荐算法应该具备模块化设计，便于扩展新的知识类型和算法类型。系统应该能够方便地添加新的推荐算法，进行A/B测试，选择最优的算法。

#### 3.2.5 可靠性需求

可靠性需求是指系统在容错性、恢复性、数据一致性等方面需要满足的要求。容错性方面，系统应该具备一定的容错能力，在出现错误时能够优雅降级，不会导致整个系统崩溃。系统应该对外部依赖进行容错处理，如数据库连接失败、第三方服务不可用等。

恢复性方面，系统应该具备快速恢复的能力，在系统异常时能够快速恢复正常运行。核心功能的宕机恢复时间应该不超过30分钟，非核心功能的恢复时间应该不超过2小时。系统应该定期备份数据，在数据丢失时能够快速恢复。数据一致性方面，系统应该保证数据的一致性，防止数据冲突和数据丢失。数据库操作应该使用事务，确保数据操作的原子性。在并发访问时，系统应该使用锁机制，防止数据冲突。

日志记录方面，系统应该记录详细的日志，包括应用日志、错误日志、安全日志等。日志应该包括时间、级别、模块、消息等信息，便于问题排查和系统监控。

## 4 系统设计

### 4.1 系统架构设计

#### 4.1.1 分层架构设计

系统采用分层设计思路，主要包括表示层、业务逻辑层、数据访问层。表示层负责用户界面展示和交互，业务逻辑层处理核心业务流程，数据访问层负责与数据库交互。系统架构设计遵循高内聚、低耦合的原则，各层之间通过明确的接口进行通信，保证了系统的可维护性和可扩展性。

分层架构的优势在于关注点分离，各层专注于自己的职责，代码结构清晰，便于理解和维护。代码复用方面，各层的组件可以独立复用，提高了开发效率。便于测试方面，各层可以独立测试，提高了代码的质量。便于扩展方面，各层可以独立扩展，不会影响其他层。

#### 4.1.2 表示层设计

表示层是系统与用户交互的界面层，负责展示数据和接收用户输入。表示层的主要职责包括页面渲染，根据业务逻辑层返回的数据渲染HTML页面；用户交互，接收用户的输入，如表单提交、按钮点击等；数据验证，在客户端对用户输入进行初步验证，提高用户体验；异步通信，使用AJAX技术实现页面的局部刷新，提升用户体验。

表示层采用Bootstrap 5实现响应式设计，确保在不同设备上都能良好显示。使用Font Awesome 6提供图标，增强页面的美观性和直观性。使用原生JavaScript实现交互效果，如表单验证、动态DOM操作等。使用Fetch API实现AJAX异步通信，实现页面的局部刷新。

#### 4.1.3 业务逻辑层设计

业务逻辑层是系统的核心层，负责处理核心业务流程。业务逻辑层的主要职责包括业务规则处理，实现系统的业务规则，如校园邮箱验证、密码强度验证、推荐算法等；流程控制，控制业务流程的执行顺序，如交易流程、消息流程等；数据转换，将数据访问层返回的数据转换为表示层需要的格式；安全验证，对用户的操作进行安全验证，确保操作的合法性。

业务逻辑层采用Django的视图（View）实现，视图接收HTTP请求，执行业务逻辑，调用数据访问层获取数据，然后将数据传递给表示层进行渲染。业务逻辑层还包括推荐算法模块、安全验证模块等，这些模块封装了核心的业务逻辑，提供清晰的接口供视图调用。

#### 4.1.4 数据访问层设计

数据访问层是系统与数据库交互的层，负责数据的存取。数据访问层的主要职责包括数据存取，提供数据的增删改查功能；数据验证，在数据存取前进行验证，确保数据的合法性；事务管理，管理数据库事务，确保数据操作的原子性；查询优化，优化数据库查询，提高查询效率。

数据访问层采用Django的ORM（对象关系映射）实现，开发者通过操作Python对象来实现对数据库的增删改查操作，Django ORM会自动将这些操作转换为对应的SQL语句。数据访问层还包括模型（Model）定义，模型对应数据库中的表，定义了数据的结构和验证规则。

### 4.2 数据库设计

#### 4.2.1 概念结构设计

概念结构设计是数据库设计的第一步，主要是建立概念模型，通常使用E-R图（实体-关系图）来表示。E-R图包含实体、属性、关系三个基本元素。

本系统的主要实体包括用户（User），属性包括用户ID、用户名、密码、邮箱、昵称、头像、手机号、个人简介、创建时间、最后登录时间等；商品（Product），属性包括商品ID、标题、描述、价格、图片、状态、浏览量、发布者、分类、创建时间、更新时间等；分类（Category），属性包括分类ID、分类名称、分类描述、父分类、创建时间等；收藏（Favorite），属性包括收藏ID、用户、商品、创建时间等；消息（Message），属性包括消息ID、发送者、接收者、内容、是否已读、创建时间等；评价（Review），属性包括评价ID、交易、评价者、被评价者、评分、评论、创建时间等；交易（Transaction），属性包括交易ID、商品、买家、卖家、交易方式、交易地点、联系电话、状态、创建时间、更新时间等；用户行为（UserBehavior），属性包括行为ID、用户、行为类型、行为时间、行为数据等；异常行为（AbnormalBehavior），属性包括异常ID、用户、异常类型、异常时间、异常数据、处理状态等；安全日志（SecurityLog），属性包括日志ID、用户、操作类型、操作时间、操作结果、IP地址、设备信息等。

实体之间的关系包括用户与商品的一对多关系，一个用户可以发布多个商品，一个商品属于一个用户；用户与收藏的一对多关系，一个用户可以收藏多个商品，一个收藏属于一个用户；商品与收藏的一对多关系，一个商品可以被多个用户收藏，一个收藏对应一个商品；用户与消息的一对多关系，一个用户可以发送多个消息，也可以接收多个消息；交易与评价的一对一关系，一个交易对应一个评价；用户与交易的一对多关系，一个用户可以作为买家参与多个交易，也可以作为卖家参与多个交易；商品与交易的一对一关系，一个商品对应一个交易；用户与用户行为的一对多关系，一个用户可以有多条行为记录；用户与异常行为的一对多关系，一个用户可以有多条异常行为记录；用户与安全日志的一对多关系，一个用户可以有多条安全日志记录。

#### 4.2.2 逻辑结构设计

逻辑结构设计是将概念模型转换为逻辑模型，通常使用关系模式来表示。关系模式就是表结构的定义，包括表名、列名、数据类型、约束等。在设计关系模式时，需要遵循规范化原则，消除数据冗余，确保数据的一致性和完整性。

本系统的主要数据表设计如下。

用户表（auth\_user）包含以下字段：id为主键，自增整数；password为密码，哈希加密存储；last\_login为最后登录时间，日期时间；is\_superuser表示是否为超级用户，布尔值；username为用户名，唯一，字符串；first\_name为名，字符串；last\_name为姓，字符串；email为邮箱，唯一，字符串；is\_staff表示是否为管理员，布尔值；is\_active表示是否激活，布尔值；date\_joined为注册时间，日期时间。

用户信息表（user\_profile）包含以下字段：id为主键，自增整数；user为外键，关联用户表，一对一关系；nickname为昵称，字符串；avatar为头像，字符串；phone为手机号，字符串；bio为个人简介，文本；created\_at为创建时间，日期时间；updated\_at为更新时间，日期时间。

商品表（product）包含以下字段：id为主键，自增整数；title为标题，字符串；description为描述，文本；price为价格，十进制数；image为图片，字符串；status为状态，字符串（可选值：available、trading、sold、removed）；views为浏览量，整数，默认0；seller为外键，关联用户表，一对多关系；category为外键，关联分类表，一对多关系；created\_at为创建时间，日期时间；updated\_at为更新时间，日期时间。

分类表（category）包含以下字段：id为主键，自增整数；name为名称，字符串；description为描述，文本；parent为外键，关联自身，一对多关系（支持多级分类）；created\_at为创建时间，日期时间。

收藏表（favorite）包含以下字段：id为主键，自增整数；user为外键，关联用户表，一对多关系；product为外键，关联商品表，一对多关系；created\_at为创建时间，日期时间；唯一约束为user和product组合唯一。

消息表（message）包含以下字段：id为主键，自增整数；sender为外键，关联用户表，一对多关系；receiver为外键，关联用户表，一对多关系；content为内容，文本；is\_read表示是否已读，布尔值，默认False；created\_at为创建时间，日期时间。

交易表（transaction）包含以下字段：id为主键，自增整数；product为外键，关联商品表，一对一关系；buyer为外键，关联用户表，一对多关系；seller为外键，关联用户表，一对多关系；trade\_method为交易方式，字符串；trade\_location为交易地点，字符串；contact\_phone为联系电话，字符串；status为状态，字符串（可选值：pending、confirmed、completed、cancelled）；created\_at为创建时间，日期时间；updated\_at为更新时间，日期时间。

评价表（review）包含以下字段：id为主键，自增整数；transaction为外键，关联交易表，一对一关系；reviewer为外键，关联用户表，一对多关系；reviewed\_user为外键，关联用户表，一对多关系；rating为评分，整数（1-5）；comment为评论，文本；created\_at为创建时间，日期时间。

用户行为表（user\_behavior）包含以下字段：id为主键，自增整数；user为外键，关联用户表，一对多关系；action\_type为行为类型，字符串（可选值：browse、search、favorite、create\_transaction、complete\_transaction等）；action\_time为行为时间，日期时间，自动添加；action\_data为行为数据，文本，可选；索引包括（user, action\_type）复合索引，（user, action\_time）复合索引。

异常行为表（abnormal\_behavior）包含以下字段：id为主键，自增整数；user为外键，关联用户表，一对多关系；abnormal\_type为异常类型，字符串；abnormal\_time为异常时间，日期时间；abnormal\_data为异常数据，文本；status为处理状态，字符串（可选值：pending、resolved、ignored）；created\_at为创建时间，日期时间。

安全日志表（security\_log）包含以下字段：id为主键，自增整数；user为外键，关联用户表，一对多关系，可选；operation\_type为操作类型，字符串；operation\_time为操作时间，日期时间；operation\_result为操作结果，字符串；ip\_address为IP地址，字符串；device\_info为设备信息，文本；created\_at为创建时间，日期时间。

#### 4.2.3 物理结构设计

物理结构设计是根据逻辑模型设计物理存储结构，包括选择数据库管理系统、设计数据表结构、创建索引、设计存储路径等。物理结构设计需要考虑性能、存储空间、维护成本等因素。

本系统选择SQLite作为数据库管理系统，SQLite将所有数据存储在单个文件中，非常适合开发阶段和小型应用。数据库文件存储在项目根目录下，文件名为db.sqlite3。

数据表的物理存储结构与逻辑结构设计基本一致，主要考虑以下几点。数据类型选择方面，根据数据的特点选择合适的数据类型，如整数、字符串、文本、日期时间、十进制数等。约束设置方面，设置主键约束、外键约束、唯一约束、非空约束等，确保数据的完整性和一致性。默认值设置方面，为某些字段设置默认值，如浏览量默认0，是否已读默认False等。字符集设置方面，使用UTF-8字符集，支持中文等多语言。

#### 4.2.4 索引优化设计

索引是数据库中用于提高查询效率的数据结构，合理的索引设计可以显著提高数据库的查询性能。本系统设计了以下索引。

用户表索引包括username唯一索引，用于用户登录时快速查找用户；email唯一索引，用于邮箱验证和密码找回时快速查找用户。

商品表索引包括（status, created\_at）复合索引，用于按状态和时间查询商品；（category, status）复合索引，用于按分类和状态查询商品；seller普通索引，用于查询某个用户发布的商品。

收藏表索引包括（user, product）唯一索引，用于防止重复收藏，同时用于查询某个用户的收藏列表；product普通索引，用于查询某个商品的收藏数。

消息表索引包括（receiver, is\_read, created\_at）复合索引，用于查询某个用户的未读消息；sender普通索引，用于查询某个用户发送的消息。

交易表索引包括buyer普通索引，用于查询某个用户作为买家的交易；seller普通索引，用于查询某个用户作为卖家的交易；product唯一索引，用于查询某个商品对应的交易。

用户行为表索引包括（user, action\_type）复合索引，用于查询某个用户的某种类型的行为；（user, action\_time）复合索引，用于查询某个用户最近的行为。

异常行为表索引包括（user, status）复合索引，用于查询某个用户的未处理异常；abnormal\_time普通索引，用于按时间查询异常行为。

安全日志表索引包括（user, operation\_time）复合索引，用于查询某个用户的安全日志；operation\_type普通索引，用于按操作类型查询安全日志。

#### 5.1.2 数据采集机制

数据采集机制实现用户行为的自动记录，在用户执行关键操作时自动触发数据采集。数据采集的主要步骤包括行为触发、数据收集、数据验证和数据存储。

### 5.2 推荐算法实现

推荐算法是系统的核心功能之一，本系统实现了多维度推荐算法，综合考虑浏览量、收藏数、交易状态、时间衰减等多个因素。

#### 5.2.1 推荐算法框架

推荐算法框架采用混合推荐策略，结合基于内容的推荐和协同过滤推荐。热门推荐算法基于商品的热度指标，包括浏览量、收藏数、发布时间等因素。个性化推荐算法基于用户的历史行为数据，分析用户的偏好特征，推荐符合用户兴趣的商品。

#### 5.2.2 特征工程设计

特征工程是推荐算法的基础，本系统设计了多维度的商品特征。浏览量特征反映商品的关注度，收藏数特征反映商品的受欢迎程度，发布时间特征反映商品的新鲜度，分类特征反映商品的类别属性。这些特征通过加权计算，生成商品的综合推荐得分。

#### 5.2.3 多维度评分机制

多维度评分机制综合考虑多个因素，计算商品的推荐得分。浏览量得分根据商品浏览量计算，最高30分；收藏数得分根据商品收藏数计算，最高25分；时间得分根据商品发布时间计算，新商品得分更高；基础得分确保所有商品都有推荐机会。总得分是各个维度得分的加权和。

#### 5.2.4 时间衰减因子实现

时间衰减因子用于缓解"冷启动"问题，新发布的商品获得更高的推荐权重。发布时间在7天内的商品获得15分的时间得分，发布时间在7-30天的商品时间得分逐渐降低，发布时间超过30天的商品时间得分为0。这种设计确保新商品能够快速获得曝光，同时避免旧商品长期占据推荐位。

#### 5.2.5 个性化推荐策略

个性化推荐策略基于用户的历史行为数据，分析用户的偏好特征。系统记录用户的浏览、搜索、收藏等行为，提取用户偏好的商品类别。推荐时优先推荐用户偏好的类别商品，同时结合商品的热度指标，生成个性化推荐列表。

**推荐算法核心实现代码**

```python
def get_hot_recommendations(request):
    """基于多维度算法计算热门推荐商品"""
    now = timezone.now()
    
    products_with_favorites = Product.objects.annotate(
        favorite_count=Count('favorite', distinct=True)
    )
    
    hot_products = []
    for product in products_with_favorites.filter(status='available')[:20]:
        views_score = min(product.views / 10, 30) if product.views else 0
        favorite_score = min(product.favorite_count * 5, 25)
        
        days_since_created = (now - product.created_at).days
        if days_since_created <= 7:
            time_score = 15
        elif days_since_created <= 30:
            time_score = 15 * (1 - (days_since_created - 7) / 23)
        else:
            time_score = 0
        
        total_score = views_score + favorite_score + 20 + time_score + 5
        hot_products.append({'product': product, 'score': total_score})
    
    hot_products.sort(key=lambda x: x['score'], reverse=True)
    return {'hot_recommendations': hot_products[:4]}
```

### 5.3 安全架构实现

安全架构是系统稳定运行的重要保障，本系统实现了三层安全架构，包括身份安全、行为安全和数据安全。

#### 5.3.1 三层安全架构设计

三层安全架构包括身份安全层、行为安全层和数据安全层。身份安全层负责用户身份验证，确保用户身份的真实性；行为安全层负责异常行为检测，防范恶意操作；数据安全层负责数据加密保护，防止数据泄露。三层架构相互配合，构建全方位的安全防护体系。

#### 5.3.2 身份安全机制

身份安全机制通过校园邮箱认证确保用户身份的真实性。系统验证用户邮箱的域名，只允许校园邮箱注册。密码强度验证要求密码包含大小写字母、数字和特殊字符，长度至少8位。双因素认证提供额外的安全保护，用户可以选择启用双因素认证提高账户安全性。

#### 5.3.3 行为安全机制

行为安全机制通过异常检测防范恶意操作。系统记录用户的登录行为，检测异常登录模式，如短时间内频繁登录、异地登录等。系统监控用户的操作行为，检测异常操作模式，如频繁发布违规商品、恶意评价等。检测到异常行为时，系统发出警告或采取限制措施。

#### 5.3.4 数据安全机制

数据安全机制通过加密技术保护用户数据。密码使用哈希算法加密存储，不能明文存储。敏感数据在传输过程中使用HTTPS加密。用户输入进行消毒处理，防止XSS攻击。SQL注入防护通过ORM查询实现，防止SQL注入攻击。CSRF防护通过Token验证实现，防止跨站请求伪造攻击。

**安全架构核心实现代码**

```python
class SecurityValidator:
    """安全验证器 - 三层安全架构核心"""
    
    @staticmethod
    def validate_email_domain(email):
        """身份安全：验证校园邮箱域名"""
        allowed_domains = getattr(settings, 'ALLOWED_EMAIL_DOMAINS', ['edu.cn', 'edu'])
        try:
            domain = email.split('@')[1].lower()
            return any(domain.endswith(allowed) for allowed in allowed_domains)
        except:
            return False
    
    @staticmethod
    def validate_password_strength(password):
        """身份安全：验证密码强度"""
        if len(password) < 8:
            return False, "密码长度至少需要8位"
        if not re.search(r'[a-z]', password):
            return False, "密码需要包含小写字母"
        if not re.search(r'[A-Z]', password):
            return False, "密码需要包含大写字母"
        if not re.search(r'\d', password):
            return False, "密码需要包含数字"
        return True, "密码强度合格"
    
    @staticmethod
    def sanitize_user_input(input_data):
        """数据安全：用户输入消毒，防止XSS攻击"""
        if not input_data:
            return input_data
        
        input_str = str(input_data)
        dangerous_tags = ['<script>', '</script>', '<iframe>', '</iframe>']
        for tag in dangerous_tags:
            input_str = input_str.replace(tag, '')
        
        input_str = input_str.replace('&', '&amp;')
        input_str = input_str.replace('<', '&lt;')
        input_str = input_str.replace('>', '&gt;')
        
        return input_str
```

## 7 总结与展望

### 7.1 总结

本研究设计并实现了一个基于Django的校园二手交易系统，重点解决了三个核心问题：构建了结构化的用户行为知识库，为智能服务提供数据基础；提出了融合多维度特征的个性化推荐算法，提升了交易匹配效率；建立了三层安全架构，保障了用户数据和交易过程的安全性。

系统测试表明，各项功能运行稳定，推荐算法效果良好，安全机制有效，为促进校园资源循环利用提供了可行的技术方案。本研究的创新点在于将知识图谱技术、推荐算法和安全机制应用于校园二手交易场景，为个性化推荐系统设计提供了新方案。
